Abstract

EduPrime is a full-stack Student Management System designed to digitize and streamline the academic administration process for educational institutions. The system provides two distinct role-based portals — one for teachers and one for students — each with tailored functionality and access controls.

The application enables teachers to manage student enrollment through a department-based join request system, track attendance, record marks, post announcements, and manage fee records. Students can register, select their department, submit a join request to their department teacher, and upon approval, access their academic records including marks, attendance, announcements, and fee status.

The project is built using modern web technologies with a Node.js and Express.js backend, AWS DynamoDB as the NoSQL database, and a responsive dark-themed frontend. The entire infrastructure is provisioned using Terraform as Infrastructure as Code (IaC), containerized with Docker, and deployed on AWS EC2 with an automated CI/CD pipeline powered by Jenkins and GitHub Webhooks.

Key features include JWT-based authentication, bcrypt password hashing, department-based access control, real-time join request polling, automatic mentor assignment, and a fully automated deployment pipeline that triggers on every GitHub push.

Chapter 1: Introduction

1.1 Overview

Educational institutions handle large volumes of student data on a daily basis — enrollment records, attendance, marks, fee payments, and announcements. Managing this data manually through paper records or disconnected spreadsheets leads to inefficiency, data loss, and poor communication between teachers and students.

EduPrime is a web-based Student Management System that addresses these challenges by providing a centralized, secure, and role-based digital platform. The system is designed to be beginner-friendly while incorporating industry-standard technologies and practices used in real-world production environments.

1.2 Motivation

The motivation behind this project is to build a production-ready application that demonstrates:

  • Full-stack web development skills (Node.js, Express, HTML/CSS/JS)
  • Cloud database integration (AWS DynamoDB)
  • Infrastructure automation (Terraform)
  • Containerization (Docker and Docker Compose)
  • Automated deployment (Jenkins CI/CD with GitHub Webhooks)
  • Secure authentication (JWT tokens and bcrypt password hashing)

1.3 Scope

The system covers the following scope:

  • User registration and authentication for students and teachers
  • Department-based student enrollment with teacher approval workflow
  • Academic record management (marks, attendance, fees, announcements)
  • Automated cloud infrastructure provisioning using Terraform
  • Continuous integration and deployment pipeline using Jenkins
  • Containerized deployment on AWS EC2

Chapter 2: Problem Statement

Traditional student management in educational institutions faces several challenges:

  • Manual Record Keeping: Attendance, marks, and fee records are maintained in physical registers or disconnected Excel sheets, making retrieval and analysis time-consuming and error-prone.
  • No Role Separation: There is no clear digital separation between what a teacher can do and what a student can access, leading to data integrity issues.
  • No Enrollment Control: Students can be added to any class without teacher approval, leading to incorrect department assignments.
  • No Automated Deployment: Code changes require manual server updates, increasing downtime and human error risk.
  • No Cloud Scalability: On-premise databases cannot scale automatically with growing student data volumes.
  • Security Gaps: Passwords stored in plain text, no token-based session management, no audit trail.

Chapter 3: Objectives

The primary objectives of EduPrime are:

  1. Role-Based Access Control — Separate portals for teachers (full CRUD) and students (read-only access to own data)
  2. Department-Based Enrollment — Students join a department and require teacher approval before accessing the portal
  3. Secure Authentication — JWT tokens and bcrypt password hashing (10 salt rounds)
  4. Cloud Database — AWS DynamoDB for scalable, serverless NoSQL storage with IAM role access
  5. Infrastructure as Code — Terraform to provision all AWS resources (DynamoDB tables, IAM roles)
  6. Containerization — Docker and Docker Compose for consistent, reproducible environments
  7. Automated CI/CD — Jenkins pipeline triggered by GitHub webhooks on every push
  8. Responsive UI — Luxury dark-themed interface accessible on all screen sizes

Chapter 6: Module Description

6.1 Authentication Module

The authentication module handles user registration and login for both students and teachers using email-based credentials.

Registration Flow:

  • User fills Full Name, Email, Password, Confirm Password
  • Student selects Department (CSE, ECE, IT, EEE, AIDS, AIML, MECH, CIVIL, CHEM, MBA, MCA)
  • Teacher enters secret Teacher Code and selects Department
  • Password hashed using bcrypt (10 salt rounds) before storage
  • User record saved to sms-users DynamoDB table
  • Teacher profile also saved to sms-teachers table
  • JWT token returned on successful registration

Login Flow:

  • Email, password, and role submitted to /api/auth/login
  • User fetched from DynamoDB by email
  • bcrypt.compare() validates password against stored hash
  • JWT token signed with 7-day expiry and returned
  • Session data stored in browser sessionStorage

6.2 Teacher Portal Module

The teacher portal provides full CRUD access to all academic data within the teacher's department.

  • Students Section: View accepted students, see pending join requests with Accept/Reject buttons, manually add or delete students
  • Join Request Management: Accept creates Student profile and assigns mentor; Reject deletes student's User account and all related data
  • Marks Section: Add, update, delete marks filtered by teacher's department
  • Attendance Section: Mark Present/Absent/Late for students with date selection
  • Announcements Section: Post and delete department announcements
  • Fees Section: Track fee payment status (Paid/Pending/Overdue)

6.3 Student Portal Module

The student portal provides read-only access to the student's own academic records.

  • Join Request Flow: On first login, system automatically sends a join request to the department teacher. Student sees a waiting screen with real-time polling every 5 seconds
  • On Acceptance: Dashboard unlocks, mentor assigned, student ID shown in sidebar
  • On Rejection: Account deleted, student must re-register with correct department
  • My Marks: Filtered to show only this student's records with grade labels (A/B/C/F)
  • Attendance: Filtered by student name with stats (Present/Absent/Late counts and percentage)
  • Announcements: All department announcements visible
  • Fees: Fee payment status with total paid and amount due stats

Chapter 7: Database Design — AWS DynamoDB

7.1 Why DynamoDB

DynamoDB was chosen over traditional relational databases for the following reasons:

  • Serverless: No database server to manage or maintain
  • Auto-scaling: PAY_PER_REQUEST billing scales automatically with usage
  • High availability: Built-in replication across AWS availability zones
  • IAM Integration: Secure access via EC2 IAM roles without hardcoded credentials

7.2 Table Designs

Table NameHash KeyGSIKey Attributes
sms-usersemail (S)—fullname, password, role, department
sms-teachersid (S)department-index, email-indexname, email, department
sms-studentsid (S)email-indexname, email, department, mentorId, mentorName, status
sms-join-requestsid (S)studentEmail-indexstudentName, studentEmail, department, status, requestedAt
sms-marksid (S)—student, subject, score
sms-attendanceid (S)—student, date, status, department
sms-announcementsid (S)—title, message, createdAt
sms-feesid (S)—student, amount, status, department

7.3 Global Secondary Indexes (GSI)

GSIs allow querying DynamoDB by non-primary-key attributes efficiently:

  • sms-teachers → department-index: Find all teachers in a specific department
  • sms-teachers → email-index: Find teacher by email address
  • sms-students → email-index: Find student profile by email
  • sms-join-requests → studentEmail-index: Find all requests by a student's email

Chapter 9: Infrastructure as Code — Terraform

9.1 What is Terraform

Terraform is an open-source Infrastructure as Code (IaC) tool by HashiCorp that allows infrastructure to be defined in human-readable configuration files and provisioned automatically. Instead of manually creating AWS resources through the console, Terraform creates, updates, and destroys resources through code — making infrastructure reproducible, version-controlled, and automated.

9.2 Terraform File Structure

FilePurpose
terraform/main.tfAll 8 DynamoDB table definitions
terraform/variables.tfInput variables (region, prefix, tags)
terraform/outputs.tfOutput values (table names after creation)
terraform/iam.tfIAM role, policy, and instance profile for EC2
terraform/.terraform.lock.hclProvider version lock file

9.3 DynamoDB Table Definition (Example)

resource "aws_dynamodb_table" "teachers" {
  name         = "sms-teachers"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "id"

  attribute { name = "id"         type = "S" }
  attribute { name = "department" type = "S" }
  attribute { name = "email"      type = "S" }

  global_secondary_index {
    name            = "department-index"
    hash_key        = "department"
    projection_type = "ALL"
  }
}

9.4 IAM Role for EC2 (No Hardcoded Keys)

resource "aws_iam_role_policy" "dynamodb_access" {
  policy = jsonencode({
    Statement = [{
      Effect   = "Allow"
      Action   = ["dynamodb:PutItem","dynamodb:GetItem",
                  "dynamodb:DeleteItem","dynamodb:Scan","dynamodb:Query"]
      Resource = "arn:aws:dynamodb:ap-south-1:*:table/sms-*"
    }]
  })
}

Chapter 10: CI/CD Pipeline — Jenkins + GitHub Webhooks

10.1 Pipeline Overview

The CI/CD pipeline automates the entire build and deployment process. Every time code is pushed to the GitHub repository, the pipeline automatically builds Docker images and deploys the updated application to the EC2 server without any manual intervention.

10.2 Pipeline Stages

StageActionDescription
1. Checkoutgit pullPull latest code from GitHub main branch
2. Build Docker Imagesdocker-compose buildBuild backend (Node.js) and frontend (Nginx) images
3. Deploy Containersdocker-compose up -dStop old containers, free ports, start new containers
4. Health Checkcurl /api/teachersVerify backend is responding correctly

10.3 Jenkinsfile (Declarative Pipeline)

The pipeline is defined in a Jenkinsfile at the root of the repository using Jenkins Declarative Pipeline syntax. When a developer pushes code to GitHub, the webhook triggers Jenkins which reads this file and executes each stage automatically.

The pipeline has 4 stages that run in sequence:

StageCommandWhat it does
1. Checkoutcheckout scmPulls the latest code from the main branch of the GitHub repository into the Jenkins workspace
2. Build Docker Imagesdocker-compose buildBuilds the backend (Node.js + Express) and frontend (Nginx) Docker images from their Dockerfiles
3. Deploy Containersdocker-compose up -dStops old containers, frees ports 3000 and 80 using fuser, then starts fresh containers in detached mode
4. Health Checkcurl /api/teachersWaits 15 seconds then sends an HTTP request to verify the backend is running and responding correctly

Post Actions run after all stages complete regardless of success or failure:

  • Success: Logs a deployment success message confirming EduPrime is live
  • Failure: Runs docker-compose down to clean up broken containers and logs the failure for debugging
  • Always: Runs docker image prune -f to remove unused Docker images and free up disk space on the EC2 server

The githubPush() trigger in the pipeline ensures that every git push to the repository automatically starts a new build — making the entire deployment process fully automated with zero manual steps required on the server.

Chapter 11: Security Implementation

11.1 Password Security — bcrypt

All passwords are hashed using bcryptjs with 10 salt rounds before storage in DynamoDB. Plain text passwords are never stored, logged, or transmitted after the initial registration request.

const hashed = await bcrypt.hash(password, 10);   // Registration
const match  = await bcrypt.compare(password, stored); // Login

11.2 JWT Authentication

JSON Web Tokens are used for stateless authentication. Tokens are signed with a secret key stored in environment variables and expire after 7 days. The token payload contains only email, role, and department — no sensitive data like passwords.

11.3 Role-Based Access Control (RBAC)

The requireTeacher middleware checks the X-Role header on every write request. Students can only perform GET requests. All POST, PUT, DELETE operations require teacher role and return HTTP 403 if unauthorized.

11.4 Teacher Registration Code

Teacher accounts require a secret code during registration, preventing unauthorized teacher account creation. The code is stored as an environment variable, not hardcoded in source code.

11.5 AWS IAM Security (Least Privilege)

  • EC2 uses an IAM Role — no hardcoded AWS credentials anywhere in code
  • IAM policy follows least privilege — only sms-* tables are accessible
  • GSI access explicitly included in the IAM policy
  • .env file is in .gitignore — never committed to GitHub
  • Terraform state files excluded from version control

11.6 Input Validation

All API endpoints validate required fields and return appropriate HTTP status codes with descriptive error messages:

HTTP CodeMeaningExample
400Bad RequestMissing required field
401UnauthorizedWrong password
403ForbiddenStudent trying to POST
404Not FoundRecord ID not found
409ConflictEmail already registered
500Server ErrorDynamoDB connection issue

11.7 Session Management

User sessions are stored in browser sessionStorage (not localStorage) so they are automatically cleared when the browser tab is closed. Logout explicitly calls sessionStorage.clear() and redirects to the login page using window.location.replace() to prevent back-button access.
